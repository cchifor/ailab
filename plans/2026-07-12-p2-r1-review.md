# Implementation review — agentforge-v2 P2 R-1 (sandbox boundary core) — round 1
<!-- codex-r1-review-status: pending -->

## Summary

- R-1 does not yet hold as a self-contained boundary: the rendered orchestrator cannot successfully create an admitted, writable sandbox Job with the current cross-repo wiring.
- Biggest-risk call: `_quiesce()` times out successfully and proceeds to import while a sandbox writer may still exist. That makes the hostile filesystem validator’s check/open gaps directly reachable.
- Admission correctly pins Kata, restricted security contexts, one container, no init containers, tokenless SA, digest-form images, allowed volume types, workspace subPath equality, and major host fields. It does not pin the complete container/pod shape, notably secret-backed environment references or resources.
- The importer covers most requested static rejection cases, including symlinks, hardlinks, special files, `.git`, collisions, size/depth, special mode bits, and schema validation. Its metadata checks are not bound to the file descriptor ultimately copied, and several limits fail open under mutation.
- Trusted Git is materially sound: Git runs only in orchestrator-controlled checkouts with clean HOME, disabled system/global config, empty hooksPath, argv execution, and no imported `.git`. Imported `.gitattributes`/`.gitconfig` cannot install executable filters or hooks under that configuration.
- RBAC is Jobs-only; `jobs/status:get` and `pods/log:get` are safe read-only additions. The active NetworkPolicy set gives test/setup Jobs zero egress, including DNS, with no additive allow policy.
- The reaper is excluded from the applied kustomization and would be rejected by sandbox admission and denied API egress if enabled. Its lease clock and deletion algorithm also race live importers.

## Findings

### Rendered configuration cannot create a valid shared-filesystem sandbox Job

**Location:** agentforge-platform/src/agentforge_platform/adapters/gitops/renderer.py:350; agentforge-platform/src/agentforge_platform/adapters/gitops/renderer.py:359; agentforge/src/agentforge/infra/settings.py:61; agentforge/src/agentforge/infra/settings.py:72; agentforge/src/agentforge/main.py:68; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-job-guard.yaml:35; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-guard.yaml:84; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-guard.yaml:94; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/workspace-pvc.yaml:13

**Severity:** blocker

<!-- codex: critique+fix -->

The renderer emits `AF_SANDBOX_SA`, but `Settings.sandbox_service_account` reads `AF_SANDBOX_SERVICE_ACCOUNT`; the application therefore uses its `af-sandbox` default, while both VAPs require `agentforge-sandbox`. Job CREATE is rejected.

The renderer also omits `AF_SANDBOX_WORKSPACE`, leaving label `default` while the configured claim is `af-sbx-ws-playground`; the pod guard consequently expects `af-sbx-ws-default` and rejects the pod. The deadline/TTL/lease names are likewise mismatched: rendered `AF_SANDBOX_JOB_DEADLINE_S`, `AF_SANDBOX_JOB_TTL_S`, and `AF_SANDBOX_LEASE_TTL_S` do not populate `sandbox_active_deadline_s`, `sandbox_ttl_after_finished_s`, or `lease_ttl_s`.

More fundamentally, the orchestrator Deployment receives no staging-root volume mount at all. The PVC exists only in `agentforge-sandbox`, while the orchestrator Pod is in a tenant namespace; PVC references are namespace-scoped. Thus staging writes go to the orchestrator container filesystem and the Job’s PVC subPath is absent. `AF_LEASE_HMAC_KEY` is also required at startup but is not explicitly rendered or coupled to the reaper key.

Define one tested environment contract using the actual `Settings` field names, render the workspace and staging root, and implement a storage topology that genuinely exposes the same backing directory to orchestrator, sandbox, and reaper across namespaces. Explicitly provision/project the same R-1 lease key. Add a cross-repo test that instantiates `Settings` from the rendered environment and validates the resulting Job against both VAP shapes.

The NFS caveat is also unresolved: staging uses ordinary `mkdir`/`copy2` without chowning or normalizing ownership, while the PVC notes that NFS ignores `fsGroup` and the sandbox runs as 65532 (`agentforge/src/agentforge/adapters/exec/sandbox.py:160`, `ailab/.../workspace-pvc.yaml:9`). A comment requesting permission verification is not sufficient to ensure the sandbox can modify the staged tree.

### Sandbox admission permits secret injection and unbounded resource shapes

**Location:** ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-guard.yaml:40; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-guard.yaml:89; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-guard.yaml:104; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-guard.yaml:118; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-job-guard.yaml:32; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/kustomization.yaml:9

**Severity:** blocker

<!-- codex: critique+fix -->

The policy constrains volume-backed Secrets but does not constrain `containers[].env`, `envFrom`, or `env[].valueFrom`. A deviating Job can therefore request any Secret in the sandbox namespace through `secretKeyRef`/`envFrom`; kubelet resolves it without the sandbox SA having Secret RBAC. Once the required reaper lease Secret is enabled, this directly exposes the lease HMAC to the sandbox and makes leases forgeable.

Container resources are entirely unconstrained, and no ResourceQuota or LimitRange is applied. A Job with absent or excessive requests/limits still passes admission, defeating the node-DoS part of the R-1 boundary.

Other unpinned fields include `command`/`args`, lifecycle, stdin/TTY, `volumeDevices`, emptyDir medium/size limits, mount flags, `imagePullSecrets`, annotations and additional labels, DNS settings, schedulerName, nodeSelector/affinity/tolerations, priority/preemption, readiness gates, and pod sysctls. Restricted PSA makes several of these non-escalating, and RuntimeClass supplies the required agent-pool selector/toleration, but the policy is not the claimed exact shape.

The Job guard only repeats runtimeClass, SA, token mounting, and restartPolicy; it does not mirror the pod guard’s template constraints. The eventual Pod guard stops most malformed templates from running, but not the environment/resource gaps above.

Forbid `envFrom` and all `valueFrom` sources for R-1, require exact resource bounds and required home/workspace volumes, pin boundary-relevant pod fields, add namespace ResourceQuota/LimitRange, and make the Job-template validation equivalent to the Pod validation.

### Quiescence fails open into hostile import

**Location:** agentforge/src/agentforge/adapters/exec/sandbox.py:456; agentforge/src/agentforge/adapters/exec/sandbox.py:460; agentforge/src/agentforge/adapters/exec/sandbox.py:475; agentforge/src/agentforge/adapters/exec/sandbox.py:515; agentforge/src/agentforge/adapters/exec/sandbox.py:525

**Severity:** blocker

<!-- codex: critique+fix -->

`_quiesce()` returns normally when its deadline expires, even if the Job or labeled Pods still exist. The caller immediately imports and applies the tree. The `finally` block similarly removes the job directory after the same unproven quiescence.

A stuck termination, node partition, kubelet failure, or malicious process blocked in filesystem I/O can therefore retain an RW NFS writer while the credential-bearing orchestrator validates and copies the directory. This violates the central “no writer remains” premise and enables concurrent type/content swaps during import.

A quiescence timeout must abort without importing or deleting the shared directory. Use foreground Job deletion followed by watch-based proof that both Job and all labeled Pods are absent; treat watch closure/resource-version loss as retry or failure, not success. Keep a valid lease until import completes.

### Import checks are not bound to the descriptor copied

**Location:** agentforge/src/agentforge/adapters/exec/import_validator.py:409; agentforge/src/agentforge/adapters/exec/import_validator.py:415; agentforge/src/agentforge/adapters/exec/import_validator.py:433; agentforge/src/agentforge/adapters/exec/import_validator.py:445; agentforge/src/agentforge/adapters/exec/import_validator.py:482; agentforge/src/agentforge/adapters/exec/import_validator.py:528; agentforge/src/agentforge/adapters/exec/sandbox.py:181; agentforge/src/agentforge/adapters/exec/sandbox.py:430

**Severity:** important

<!-- codex: critique+fix -->

The Linux walker performs `DirEntry.stat(follow_symlinks=false)`, validates that result, and then separately opens the name. `O_NOFOLLOW` prevents a symlink substitution, but there is no `fstat()` of the opened descriptor. A concurrent regular-to-regular, hardlink, FIFO, or device swap can therefore bypass the validated type, owner, link count, xattrs, and initial size. Directory metadata and xattrs are similarly obtained through pathname operations rather than the opened directory descriptor.

Actual bytes copied are checked only against the per-file cap; cumulative size accounting uses the earlier `stat` size. A tree can grow after accounting and exceed the 256 MiB total. Directories are not counted at all, so an arbitrarily wide empty-directory tree bypasses `max_files`. `listxattr` errors are treated as “no xattrs,” and the Windows degraded path has neither descriptor-relative traversal nor `O_NOFOLLOW`. Default wiring also leaves `expected_uid=None`, disabling the ownership rule. Finally, `.af-result.json` is read using `is_file()` and ordinary `open()`, following a hostile symlink in the orchestrator’s filesystem namespace.

After making quiescence fail closed, validate the opened object with descriptor-relative `fstat`/`flistxattr`, reject inode/type/metadata changes, account actual streamed bytes, bound directories and total entries, and open result metadata with no-follow semantics. Fail closed on xattr inspection errors and refuse the path-based degraded mode for production imports.

### The lease/reaper hand-off is neither deployed nor race-free

**Location:** ailab/kubernetes/apps/infrastructure/agentforge-sandbox/kustomization.yaml:17; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/reaper-deployment.yaml:7; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/reaper-deployment.yaml:39; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/networkpolicy.yaml:15; ailab/kubernetes/apps/infrastructure/agentforge-sandbox/sandbox-guard.yaml:154; agentforge/src/agentforge/adapters/exec/sandbox.py:433; agentforge/src/agentforge/adapters/exec/sandbox.py:446; agentforge/src/agentforge/main.py:425; agentforge/src/agentforge/adapters/exec/reaper.py:83; agentforge/src/agentforge/adapters/exec/kube.py:138

**Severity:** blocker

<!-- codex: critique+fix -->

The reaper Secret and Deployment are commented out, so R-1 has no active directory reaper. Enabling the manifest as written still does not work: the namespace-wide pod guard demands a container named `sandbox`, Kata runtime, and sandbox SA, so it rejects the `reaper` Pod; the namespace-wide zero-egress policy also prevents that Pod from reaching the Kubernetes API.

The lease itself is stamped using `time.monotonic()`, while the real reaper compares it with `time.time()`. Every such lease appears expired immediately. It is written once and never renewed while running/importing. Because the file resides in the job-writable directory, a hostile job can also delete or corrupt it; the reaper interprets that as an absent lease eligible for collection.

The reaper checks `status.active` once, then reads the lease, then deletes the Job and immediately `rmtree`s the directory without waiting for Job/Pod disappearance. A Pending Job, stale status, a Job becoming active between reads, or a terminating Pod can therefore still own a writer when the directory is removed.

Deploy the reaper in a policy/network context where its exact trusted shape and API egress are allowed, using the same genuinely shared storage. Use one wall-clock domain, atomic periodic renewal, and lease state outside the untrusted writable subtree. Reclaim must acquire ownership atomically, recheck Job and Pod state, delete foreground, wait for all writers to disappear, and only then remove the directory. A real coordination Lease should replace the always-leader stub.

## Verdict

R-1 is not yet a sound boundary to proceed: fix the deployment/wiring contract, exact admission gaps, fail-open quiescence/import path, and deployed race-free reaper before claiming the credential/node boundary.
